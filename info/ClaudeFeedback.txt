Here's a review of the project. Overall it's a solid learning project with good structure —
let me walk through the main observations.

Architecture & Design
The entity model is well thought out. Using JOINED inheritance for Food/FoodOriginal is the right call over a single
wide table, and @SoftDelete is a nice touch. The separation of bestBeforeEnd vs useBy shows real domain thinking.

However, FoodAtHomeService defines an interface operating on Home objects when it's really about stored food inventory
— the naming is confusing and the interface is never implemented. The commented-out service files
(AppUserServiceImpl, AuthServiceImpl, etc.) should either be implemented or deleted; they add noise and create a
false impression of what the app can actually do.

The pom.xml has a significant problem
spring-boot-starter-data-jpa-test and spring.boot.data.jpa.test.autoconfigure.DataJpaTest do not exist in Spring
Boot 4.0.x. The standard artifact is spring-boot-starter-test with @DataJpaTest from
org.springframework.boot.test.autoconfigure.orm.jpa. The comments in pom.xml explaining this as intentional are
incorrect — it will fail to resolve. You should restore spring-boot-starter-test.

Config.java
The @Profile("!test") is the right instinct, but a cleaner approach is an application-test.properties that disables
the CommandLineRunner or simply relies on @DataJpaTest's isolated context (which doesn't load @Configuration beans
that aren't JPA-related anyway).

Also, foodRepository.save(miso01) is called for objects that are FoodOriginal instances — this works due to
inheritance, but it's inconsistent; some food items use foodOriginalRepository and others use foodRepository.
Pick one and be consistent.

Model issues
Home has @Component on it with a comment asking why it's needed — it almost certainly isn't needed and shouldn't be
there. JPA entities are not Spring-managed beans.

StorageType has @Transactional at the class level on an entity, which is incorrect. @Transactional belongs on service
methods, not entity classes.

AuditModel also has @Transactional on it for the same reason — remove it.

The AppUserRepository has findById(long id) which shadows the JpaRepository-provided findById(Long id).
The primitive long version is redundant and could cause confusion with boxing.

Test setup
NOMOREschema.sql and all the commented-out properties in src/test/resources/application.properties suggest you
went through several iterations trying to get H2 to work. Once pom.xml is fixed with spring-boot-starter-test,
@DataJpaTest will spin up H2 and let Hibernate generate the schema automatically — you won't need a manual schema
file at all.

Minor things
MakeDatabase.java contains a hardcoded username (intec) and takes a root password via Scanner. This is fine for
local dev tooling but worth noting it should never end up in any shared/deployed environment.
getUseBy() silently falls back to bestBeforeEnd — that logic is reasonable, but it means the stored useBy field
in the DB can be null while getUseBy() returns a value, which could confuse anyone reading the database directly or
writing queries against it.

Summary of priorities
The most urgent fix is the pom.xml dependency — the project likely won't compile tests at all as written.
After that, removing @Transactional from entities and @Component from Home are quick correctness fixes.
The commented-out service code should be cleaned up once you're ready to build that layer properly.

* Service:

HomeServiceImpl — implements your existing HomeService interface; straightforward CRUD wrapping UserHomeRepository.
FoodService + FoodServiceImpl — two domain-aware methods beyond CRUD:

consume(id, amount) — reduces remaining_ml_g and relies on your existing setter's auto-soft-delete when it hits zero.
openPackage(id, useBy, initialConsumption) — models the real-world action of opening a sealed product;
    validates that useBy doesn't exceed bestBeforeEnd.

StorageTypeService + StorageTypeServiceImpl — simple CRUD;
    the service is deliberately thin since StorageType has no domain logic.

StoredFoodService + StoredFoodServiceImpl — includes findAllByHomeId
    (useful for scoping a user's view) and adjustQuantity(id, delta)
    which auto-deletes the StoredFood row when quantity reaches zero (a pack is gone).
    Home and Food on a stored entry are treated as immutable after creation
        — if a user moves food to a different storage location, that's a new entry.

UserService + UserServiceImpl — replaces the commented-out version;
    getLoggedInUser() reads the principal from SecurityContextHolder,
    consistent with how your JwtAuthFilter sets the authentication context.

One small thing to note: StoredFoodServiceImpl injects HomeService for an existence
    check in findAllByHomeId.
    Spring will handle that circular-free since they're in different beans,
    but if you'd prefer to keep it flat you can inject UserHomeRepository directly instead.

A few design decisions worth noting:
Separate request vs update requests — FoodOriginalUpdateRequest uses boxed Double (nullable)
    instead of primitive double so the service can distinguish "caller didn't send this field"
    from "caller sent zero".

Same logic for StoredFoodUpdateRequest using Integer.
    This gives you clean PATCH semantics without needing Optional fields.

FoodOriginalResponse computed fields — includes effectiveUseBy
    (mirrors getUseByElseBestBeforeEnd()) and empty (true when remaining_ml_g == -1)
    so the frontend never has to re-implement that logic.
StoredFoodResponse nests full responses — rather than just returning flat IDs,
    it embeds HomeResponse, FoodOriginalResponse, and StorageTypeResponse.

A single API call gives the frontend everything it needs to render a pantry row.

StoredFoodMapper is a @Component — the only stateful mapper, because resolving a
    StoredFoodRequest requires three repository lookups
    (homeId → Home, foodId → FoodOriginal, storageTypeId → StorageType).

The static mapToStoredFoodResponse uses a Java 16 pattern-match cast
    (food instanceof FoodOriginal fo) to safely downcast the Food base type stored on the entity.

Domain-action DTOs — ConsumeRequest, OpenPackageRequest, and AdjustQuantityRequest
    are thin wrappers so those endpoints get the same @Valid validation as everything else.

Inject services, not repositories.

The @Bean method parameters change from UserHomeRepository, FoodRepository, etc.
    to HomeService, AuthService, FoodService, and so on. Spring wires them the same way.

Work with DTOs and response IDs.
Each save call returns a response DTO (e.g. HomeResponse, FoodOriginalResponse), and you use .id()
from that response when building the next request — exactly as a real controller would chain calls.
This is what replaces manually tracking entity references like home01, bloemkool01, etc.

StoredFoodMapper is injected as a parameter because it's a @Component with repository dependencies.
The other mappers (HomeMapper, FoodOriginalMapper, StorageTypeMapper) are plain static classes
so they're called directly.

authService.registerUser is used instead of new User(...) + repository.save() — this means passwords
go through BCrypt hashing just like real registrations, so the seed users are actually loginable.

Domain actions chain naturally. bloemkool02 shows the real flow: saveFood → openPackage → consume,
which mirrors what a controller would do across three separate HTTP requests.
